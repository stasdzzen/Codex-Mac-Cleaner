# Участие в разработке

Спасибо за вклад в Codex Mac Cleaner. Проект принимает небольшие проверяемые
изменения с явной связью между проблемой, каноном и результатом.

## Перед началом

1. Прочитайте [`AGENTS.md`](AGENTS.md), [архитектурный канон](docs/index.md) и
   [контракт выполнения](docs/development/execution-contract.md).
2. Найдите существующую [задачу](https://github.com/stasdzzen/Codex-Mac-Cleaner/issues)
   или создайте сообщение об ошибке либо небольшое предложение через форму GitHub.
3. Согласуйте границы до изменений архитектуры, безопасности, переноса данных или выпуска.

Не публикуйте уязвимости в задачах или запросах на слияние. Для них действует закрытый
порядок из [SECURITY.md](SECURITY.md).

## Поддерживаемая среда

Продукт поддерживает только macOS 26 или новее на Mac с процессором Apple M1
или новее (`arm64`). Проверки репозитория используют Python 3 в GitHub Actions,
но пользователю плагина Python не нужен. Версии средств разработки закреплены
в файле зависимостей.

## Рабочий порядок

1. Одна задача GitHub ведёт к одной задаче Codex, одной рабочей копии, одной ветке и одному запросу на слияние.
2. Не расширяйте границы задачи и не меняйте архитектурный канон без ADR.
3. Для нового поведения сначала добавьте падающий тест, затем код и только после этого улучшайте структуру.
4. Не тестируйте очистку на реальном Mac. Используйте искусственные тестовые данные. Ручную проверку на настоящем Mac запускает только владелец проекта.
5. Не добавляйте секреты, полные пути к домашней папке, список настоящих приложений, персональные данные, телеметрию или необработанные настройки.
6. Пишите коммиты, запросы на слияние, отзывы и комментарии GitHub по-русски.

## Проверки

До создания запроса на слияние выполните:

```bash
python3 .github/scripts/validate_repository.py
python3 -m unittest discover -s tests -p 'test_repository_policy.py' -v
git diff --check
```

Для рабочего приложения дополнительно обязателен `pnpm check` из корня
репозитория. Проверки отдельных файлов не заменяют полный набор проверок.

## Запрос на слияние

* свяжите запрос с задачей через `Closes #...`;
* перечислите изменения и точные команды проверок;
* укажите риски, ограничения и проверки, которые может выполнить только владелец;
* не смешивайте несвязанные исправления;
* не отмечайте слияние, выпуск или ручную проверку выполненными заранее.

Шаблон запроса содержит обязательный список. Сообщение автора об успехе не
заменяет GitHub Actions и проверку текущего коммита.

## Выпуск версии

Выпуск, тег, заверение приложения Apple и публикация плагина выполняются только
после отдельного подтверждения владельца и всех проверок выпуска. Обычный
запрос на слияние не создаёт тег и не меняет опубликованную версию.
