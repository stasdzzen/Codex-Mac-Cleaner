# Codex Mac Cleaner MCP App

`SPEC.md` — краткая продуктовая спецификация MCP App для реализации. Архитектурным
каноном остаются документы из [`docs/`](docs/index.md); при расхождении действует
канон.

## Ценность

Codex Mac Cleaner помогает владельцу Mac найти доказуемые остатки удалённых
приложений и принять безопасное решение без копирования команд в терминал.
Пользователь видит объяснение, риск и blocking reason, после чего отдельно решает,
оставить один объект, переместить его в карантин, восстановить или удалить навсегда.

Основные действия:

1. Запустить read-only аудит `application_remnants`.
2. Изучить прогресс, находки и доказательства в Audit Dashboard.
3. Выполнить одно явно подтверждённое действие над одним объектом.

## Зачем здесь LLM

Пользователь может естественным языком попросить Codex провести диагностику и
получить объяснение результата. Модель помогает сформулировать намерение и объясняет
server-owned выводы, но не классифицирует файлы и не получает полномочий на mutation.
Локальные данные, policy, correlation, карантин и действия предоставляет MCP-сервер.

## Пользовательский путь

1. Пользователь просит провести диагностику.
2. Skill запускает read-only аудит и сразу открывает один view `dashboard_open`.
3. Dashboard показывает server-owned прогресс, а после завершения — immutable
   результаты в сгруппированной таблице и доступные поэлементные действия.
4. В новой задаче пользователь может явно открыть последнюю завершённую ревизию
   без повторного аудита.
5. Пользователь при необходимости нажимает «Развернуть» или «Свернуть»;
   завершение аудита не меняет выбранный режим.
6. Codex может принять или отклонить запрос режима; текущий Dashboard остаётся
   работоспособным в любом случае.
7. Пользователь изучает находку и отдельно подтверждает допустимое действие.
8. В подвале пользователь может открыть репозиторий, GitHub Ideas, сайт
   разработчика или будущую страницу поддержки проекта.

## View и tools

### View `dashboard_open`

- Вход: explicit `auditId` и nullable `revision` либо `null/null` для последней
  сохранённой completed revision.
- Выход: безопасный Dashboard snapshot и widget-only metadata.
- UI: четыре вкладки; живой прогресс и ограниченная grouped-таблица находятся на
  «Обзоре», остальные разделы — карантин, оставленные объекты и ручной запуск.
- Отображение: `inline` по умолчанию; `fullscreen` и возврат в `inline`
  запрашиваются только отдельным нажатием пользователя. Режим `pip` продукт не
  предлагает.
- Fallback: отсутствие API или отказ хоста показываются текстом и не меняют аудит.
- Ограничение: публичный MCP Apps API не гарантирует размещение в правой панели;
  плагин не заявляет и не эмулирует такой режим.
- Подвал: `© 2026 Dzzen`, GitHub, «Поделиться идеей», «Разработчик» и
  «Поддержать проект». Внешние ссылки фиксированы, открываются только по клику
  через host API и не вызывают MCP tools.
- Результаты: Dashboard v4 получает server-owned итоговые counts, не более 100
  находок и не более 512 КиБ сериализованного массива на странице. Кеши
  группируются вместе, остатки удалённых приложений — по приложению. Следующая
  widget-safe страница загружается app-only инструментом `dashboard_page` только
  после нажатия «Показать ещё».
- Cursor: непрозрачен, привязан к audit/revision/каналу/filters и не содержит
  path, identity или mutation authority. После stale cursor интерфейс сохраняет
  уже показанные данные и объясняет, что страницу загрузить не удалось.

### Tools

Состав model-visible и app-visible tools определяется
[`docs/contracts/mcp-tools.md`](docs/contracts/mcp-tools.md). Переключение режима
отображения является локальным UI-запросом к хосту и не создаёт MCP mutation-tool.
`audit_results` отдаёт model-safe страницы, а `dashboard_page` доступен только
интерфейсу и возвращает widget-safe страницу. Эти cursors не взаимозаменяемы.

## Продуктовый контекст

- Среда: Codex для macOS 26 на Apple Silicon.
- Runtime: локальный TypeScript/Node.js MCP server через stdio.
- UI: автономный React/Vite bundle без CDN и отдельного HTTP-сервера.
- Доступ: локальная установка через repository/personal marketplace.
- Безопасность: server-owned policy, protected scopes, приватный карантин,
  поэлементное подтверждение и отсутствие автоматического удаления.

## Производительность аудита

- У всего запуска нет автоматического общего лимита времени: после discovery
  сервер обрабатывает каждый найденный кандидат и только затем публикует
  завершённую immutable revision.
- Глобальные inventories, включая package inventory, снимаются не более одного раза
  для Snapshot A и одного раза для Snapshot B.
- Candidate-specific evidence повторно проверяется в обеих фазах с фиксированной
  bounded concurrency восемь; порядок итоговых findings остаётся детерминированным.
- Защитный таймаут отдельной read-only команды macOS превращает недоступный запрос
  в coverage gap, но не прерывает весь аудит.
- Оптимизация не обрезает список кандидатов, не превращает partial coverage в
  complete и не создаёт actionable revision после cancellation или internal error.
- Все byte-метрики Dashboard отображаются только в десятичных МБ/ГБ; значения
  меньше 0,01 МБ не округляются до ложного нуля.
- User missing-target LaunchAgents отображаются как `analysis_only`, системные
  missing-target LaunchAgents/LaunchDaemons — как `unsupported_manual`; оба класса
  доступны только для инспекции и не получают mutation или shell actions.
- Активный процесс с доказанно отсутствующим абсолютным executable отображается
  только как read-only diagnostic; PID и полный путь не покидают server boundary.
