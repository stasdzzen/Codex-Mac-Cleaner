# Контекст

ADR-0018 сохранил пятиминутный server-owned deadline и увеличил bounded
concurrency с четырёх до восьми, исходя из двух измерений beta.6. Реальный запуск
beta.9 на Mac владельца показал, что этой оптимизации недостаточно: аудит
продолжал стабильно продвигаться, но через пять минут был принудительно остановлен
на 2 206 из 2 767 кандидатов с `AUDIT_TIMEOUT`. Permission gap или другая
предшествующая ошибка источника не наблюдались.

Автоматическое завершение по времени противоречит основной продуктовой задаче:
пользователь запускает проверку, чтобы увидеть полный результат по всем найденным
объектам. Повтор того же запуска не устраняет детерминированное ограничение, а
частичная actionable revision ослабила бы Snapshot A/B и полноту отрицательных
доказательств.

# Решение

1. У всего audit run нет hard, soft или progress-aware deadline. Прошедшее
   wall-clock время само по себе не меняет состояние запуска и не вызывает
   `AbortController`.
2. После discovery Coordinator обрабатывает все найденные кандидаты. Immutable
   revision публикуется только после завершения полного прохода, Snapshot B и
   финализации.
3. Пользователь сохраняет явную кооперативную отмену через `audit_cancel`. Отмена
   переводит запуск в `cancelled`, не публикует частичную actionable revision и
   не открывает mutation actions.
4. Ошибка одного source query не завершает весь аудит. Она становится
   типизированным coverage gap, соответствующие факты остаются `unknown`, а
   изменяющие действия для затронутых findings блокируются.
5. Защитный таймаут отдельной системной команды macOS сохраняется. Он ограничивает
   один вызов `pkgutil`, `plutil` или другого read-only subprocess, но не является
   лимитом всего аудита и не отменяет обработку остальных кандидатов.
6. Bounded concurrency восемь, детерминированный порядок результатов, server-owned
   progress, protected scopes, Snapshot A/B, correlation requirements и mutation
   policy не меняются.
7. Текущий продуктовый код `AUDIT_TIMEOUT` удаляется: новый runtime не имеет
   состояния, которое может его выпустить. Реальная внутренняя ошибка по-прежнему
   завершает run как `failed` с безопасным `INTERNAL_ERROR`.

Решение заменяет только пункт 2 ADR-0018 и связанные утверждения о пятиминутной
границе. Остальные решения ADR-0018 остаются действующими.

# Safety-инварианты

* Долгий run не создаёт раннюю или частичную revision.
* Cancellation и internal error не сохраняют findings с mutation actions.
* Один зависший subprocess ограничен собственным таймаутом и превращается в
  fail-closed coverage gap.
* Пользователь в любой момент может явно отменить read-only аудит.
* Никакой длительный аудит не запускается автоматически или в фоне в v0.1.

# Последствия

* Полный аудит большого реального Mac может выполняться дольше пяти минут.
* Dashboard продолжает показывать фазу и честные candidate counts, поэтому
  пользователь видит, что проверка продвигается.
* Верхняя граница длительности заранее не обещается: она зависит от числа объектов,
  скорости диска и ответов macOS.
* Потребление ресурсов ограничено concurrency восемь и пооперационными таймаутами;
  окончательное решение об остановке всего запуска принадлежит пользователю.
* Ручной owner real-Mac smoke остаётся отдельным gate следующего prerelease.
