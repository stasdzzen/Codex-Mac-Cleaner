# Цель v0.1

Codex Mac Cleaner v0.1 находит остатки обычных macOS-приложений, объясняет выводы и перемещает отдельно одобренные объекты в обратимый карантин.

# Целевая платформа

* macOS 26 и новее;
* Apple Silicon, архитектура `arm64`;
* Intel и Rosetta не поддерживаются;
* совместимость со старыми версиями macOS не проектируется.

# Обязательные возможности

* read-only аудит без изменения пользовательских данных;
* структурированный отчёт о покрытии и недоступных областях;
* доказательства, контрдоказательства и недостающие сведения для каждой находки;
* метка состояния и отдельный уровень уверенности;
* серверное вычисление `allowedActions`;
* «Показать в Finder»;
* server-side protected scopes, которые нельзя ослабить через UI или конфигурацию;
* versioned identity-based persistent exclusions и UI управления ими;
* безопасные метаданные JSON/YAML/plist без сырых ключей, значений и секретов;
* три уровня поддержки находки: `candidate`, `analysis_only`, `unsupported_manual`;
* поэлементный preview и подтверждение карантина;
* Quarantine Center с поэлементным восстановлением и очисткой;
* восстановление только в исходный путь без перезаписи;
* ручной preview и подтверждение окончательной очистки;
* отмена долгого read-only аудита без actionable partial revision;
* раздельные логический и физический размеры находок, карантинный и окончательно удалённый объём;
* наблюдаемое состояние диска без обещания причинного APFS delta;
* установка, аудит и решения в интерфейсе Codex без копирования shell-команд;
* вкладка «Расписание» как честный disabled/manual-run fallback без host-native automation lifecycle;
* публичный bundle без персональных app/path rules, username и real-Mac inventory;
* локальная работа без сети и телеметрии.

# Модель доступа

Full Disk Access не обязателен. Без него продукт сканирует доступные области и возвращает структурированный список пробелов покрытия. Дополнительное разрешение может расширить покрытие, но его отсутствие не превращает весь аудит в ошибку.

# Что не входит в v0.1

* артефакты разработки: `node_modules`, `.next`, `DerivedData`, симуляторы, Docker, Colima, кэши пакетных менеджеров и Git worktrees;
* uv, npm/nvm, pnpm store, node-gyp, Puppeteer/Chromium, Whisper models, Homebrew downloads/formulae, глобальные Node/Python/uv tools, SDK и локальные developer databases;
* личные документы и поиск дубликатов по содержимому;
* внешние и сетевые тома;
* автоматическая фоновая очистка;
* создание, update, pause, resume или delete host-native Codex automation и выполнение scheduled prompt;
* собственный cron, LaunchAgent или другой скрытый scheduler;
* массовая кнопка «Очистить всё»;
* прямое удаление исходного объекта;
* `sudo`, Swift-helper, LaunchDaemon или system extension;
* изменение VPN, DNS, proxy, системных расширений и системного автозапуска;
* mutation в `/Library`, управление LaunchAgents/LaunchDaemons/PrivilegedHelperTools, APFS или Time Machine snapshots;
* облачная синхронизация, аккаунты и телеметрия.

# Принципы

1. Сначала чтение, затем отдельное решение о действии.
2. Доказательства важнее эвристики.
3. Классификация не равна разрешению на карантин.
4. Неопределённость блокирует действие, а не скрывается.
5. Одна операция относится к одному объекту верхнего уровня.
6. Модель и интерфейс не передают произвольные пути в mutation-контур.
7. Любая гонка или противоречие завершает операцию без изменения данных.
8. Восстановление — часть определения готовности, а не дополнительная функция.
9. Отменённый аудит не даёт разрешения на mutation.
10. Метрика не может обещать точный прирост свободного места APFS.
11. Совпадение имени не доказывает владельца или безопасность действия.
12. Protected scope исключается до создания кандидата и повторно проверяется перед mutation.
13. Секреты редактируются до persistence, логирования и MCP output.
14. `unsupported_manual` объясняет границу, но не содержит готовую shell-команду.
15. User exclusion не ослабляет protected scope и не совпадает только по path.
16. В v0.1 расписание не создаётся: вкладка остаётся disabled и предлагает только ручной read-only audit; host lifecycle требует отдельного post-v0.1 owner decision.
17. Полевые данные превращаются только в универсальные synthetic cases, а не в персональные runtime-правила.
